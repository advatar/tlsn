Yes. I’d do it, but I’d freeze the first target very tightly.

The repo already has the right shape for this: a workspace with dedicated `components`, `key-exchange`, `mpc-tls`, `tls/client`, `tls/core`, `tlsn`, `attestation`, `formats`, `harness`, and `server-fixture` crates; upstream also already has the meta issue for TLS 1.3 support (`#859`), a focused issue for TLS 1.3 MPC-HKDF (`#978`), and an open PR (`#1001`) that says it implements a TLS 1.3 key schedule and refactors the existing HMAC/PRF code around it.  [oai_citation:0‡GitHub](https://raw.githubusercontent.com/tlsnotary/tlsn/main/Cargo.toml)

## Freeze v0 scope

1. **Full 1-RTT only.** No PSK, no resumption, no 0-RTT. TLS 1.3 0-RTT has weaker guarantees, is not forward secret in the same way, and has explicit replay caveats in RFC 8446.  [oai_citation:1‡IETF Datatracker](https://datatracker.ietf.org/doc/html/rfc8446)

2. **`TLS_AES_128_GCM_SHA256` only.** The current workspace already has a TLS 1.2 MPC HMAC-SHA256 component, and `tlsn-mpc-tls` already pulls in `aes-gcm` and `ghash`; the recent upstream PR is also explicitly scoped to a TLS 1.3 key schedule in the HMAC-SHA256 area. The USENIX “garble-then-prove” paper also uses `AES128_GCM_SHA256` as its main concrete mode and says `AES256_GCM`/`SHA384` can be added later.  [oai_citation:2‡GitHub](https://raw.githubusercontent.com/tlsnotary/tlsn/main/crates/components/hmac-sha256/Cargo.toml)

3. **P-256 first, X25519 second.** The dedicated 3-party key-exchange crate currently depends on `p256`, so P-256 is the shortest path to a working TLS 1.3 handshake. Broad compatibility will still need X25519 soon after, but I would not block the first merge on that.  [oai_citation:3‡GitHub](https://raw.githubusercontent.com/tlsnotary/tlsn/main/crates/components/key-exchange/Cargo.toml)

4. **No client cert auth, no post-handshake auth, no KeyUpdate in v0.** RFC 8446 makes `KeyUpdate` a post-`Finished` feature, and RFC 8448 treats client authentication as a separate trace. That makes them clean follow-on milestones instead of day-one blockers.  [oai_citation:4‡IETF Datatracker](https://datatracker.ietf.org/doc/html/rfc8446)

5. **HelloRetryRequest as milestone 2, not milestone 1.** TLS 1.3 HRR has special transcript rules using the synthetic `message_hash`, so it is worth implementing deliberately after the simple 1-RTT path is green.  [oai_citation:5‡IETF Datatracker](https://datatracker.ietf.org/doc/html/rfc8446)

## Crate-by-crate plan

### 1) `crates/components/hmac-sha256`

This should be the first merge target. Right now that crate is explicitly a TLS 1.2 SHA-256 HMAC PRF component with `MpcPrf`, `Mode::{Normal,Reduced}`, and TLS 1.2-style session-key output. PR `#1001` already says it isolates HMAC functionality into a separate module and implements a TLS 1.3 key schedule on top.  [oai_citation:6‡GitHub](https://raw.githubusercontent.com/tlsnotary/tlsn/main/crates/components/hmac-sha256/src/lib.rs)

What I’d do:

- keep the existing TLS 1.2 `MpcPrf` API working
- add HKDF primitives:
  - `hkdf_extract`
  - `hkdf_expand`
  - `hkdf_expand_label`
  - `derive_secret`
  - `finished_key`
- make transcript-hash input explicit, because TLS 1.3 derives secrets from specific transcript prefixes

The target semantics come straight from RFC 8446: `HKDF-Expand-Label`, `Derive-Secret`, the `"c hs traffic"` / `"s hs traffic"` / `"c ap traffic"` labels, and the `finished_key` formula are all specified there.  [oai_citation:7‡IETF Datatracker](https://datatracker.ietf.org/doc/html/rfc8446)

### 2) `crates/components/key-exchange`

Today this crate is a 3-party ECDH exchange with a `Pms` type alias and a `MpcKeyExchange` API, and it is wired around `p256`. For TLS 1.3, the conceptual output is no longer “premaster secret” in the TLS 1.2 sense; it is just the ECDHE shared secret that feeds HKDF.  [oai_citation:8‡GitHub](https://raw.githubusercontent.com/tlsnotary/tlsn/main/crates/components/key-exchange/src/lib.rs)

I would:

- keep the current P-256 path for the first TLS 1.3 milestone
- stop leaking the TLS 1.2 name upward:
  - either rename `Pms` to `SharedSecret`
  - or add a versioned wrapper so TLS 1.2 keeps `Pms` while TLS 1.3 consumes `EcdheSecret`
- add an X25519 work item as the next compatibility milestone, not part of the first merge

### 3) `crates/mpc-tls`

This is where the biggest structural change lands. The current public surface is TLS 1.2-shaped: `MpcTlsLeader`, `MpcTlsFollower`, a `record_layer`, and `SessionKeys` containing client/server write keys, IVs, and a `server_write_mac_key`. TLS 1.3 does not look like that: it has handshake traffic secrets, application traffic secrets, AEAD-only record protection, and encrypted handshake messages after `ServerHello`.  [oai_citation:9‡GitHub](https://raw.githubusercontent.com/tlsnotary/tlsn/main/crates/mpc-tls/src/lib.rs)

I’d replace the current implicit model with explicit epochs:

```rust
enum Epoch {
    Handshake,
    Application,
}

struct TrafficKeys {
    secret: SecretBytes,
    key: SecretBytes,
    iv: SecretBytes,
    seq: u64,
}

struct Tls13Secrets {
    c_hs: TrafficKeys,
    s_hs: TrafficKeys,
    c_app: TrafficKeys,
    s_app: TrafficKeys,
    client_finished_key: SecretBytes,
    server_finished_key: SecretBytes,
}
```

Then wire `leader`/`follower` like this:

- run 3-party ECDHE as today
- run MPC-HKDF to derive both handshake and application secrets
- **reveal only handshake traffic keys to the prover/leader**
- keep application traffic keys secret-shared

That reveal pattern is the critical TLS 1.3 design choice. RFC 8446 requires the server flight after `ServerHello` to be encrypted, and DECO’s TLS 1.3 adaptation explicitly recommends revealing handshake keys to the prover while keeping final application secrets protected, because TLS 1.3’s key schedule separates them cleanly.  [oai_citation:10‡IETF Datatracker](https://datatracker.ietf.org/doc/html/rfc8446)

### 4) `crates/tls/core` and `crates/tls/client`

`tls/client` is where the client state machine lives today; its public docs and module layout include `conn`, `kx`, `record_layer`, `builder`, `cipher`, and `backend`. That is the right landing zone for the protocol logic.  [oai_citation:11‡GitHub](https://raw.githubusercontent.com/tlsnotary/tlsn/main/crates/tls/client/src/lib.rs)

For the TLS 1.3 path, add:

- version negotiation and `supported_versions`
- TLS 1.3 `key_share`
- `EncryptedExtensions`
- `Certificate`
- `CertificateVerify`
- `Finished`
- transcript helpers for:
  - `ClientHello ... ServerHello`
  - `ClientHello ... server Finished`
  - HRR synthetic `message_hash`

The key RFC facts to encode are:

- all handshake messages after `ServerHello` are encrypted
- `EncryptedExtensions` is the first encrypted server handshake message
- `finished_key` is derived with `HKDF-Expand-Label(BaseKey, "finished", "", Hash.length)`
- HRR rewrites the transcript using synthetic `message_hash(Hash(ClientHello1))`  [oai_citation:12‡IETF Datatracker](https://datatracker.ietf.org/doc/html/rfc8446)

Milestone 1 state machine:

- client sends `ClientHello`
- receives `ServerHello`
- derives handshake secrets
- decrypts server handshake locally with revealed handshake keys
- verifies `CertificateVerify`
- verifies `Finished`
- derives application secrets
- sends client `Finished`
- switches record layer to application epoch

Milestone 1 rejects:

- PSK/resumption/0-RTT
- client certificate auth
- `KeyUpdate`
- post-handshake auth

### 5) `crates/tlsn`, `attestation`, and `formats`

The top-level `tlsn` crate is about transcript commitment and selective disclosure, not the handshake mechanics themselves. That means most of the proof model can stay the same. What does need to change is the metadata and serialization so the attestation clearly says:

- TLS version = 1.3
- cipher suite
- key exchange group
- handshake mode = full 1-RTT

The transcript commitment flow itself should stay version-agnostic as much as possible. The workspace already has dedicated `attestation` and `formats` crates, so this is the right place to version the wire format cleanly instead of baking TLS 1.3 assumptions into the prover/verifier API.  [oai_citation:13‡GitHub](https://raw.githubusercontent.com/tlsnotary/tlsn/main/Cargo.toml)

## Test plan

Use both the existing local fixtures and the published RFC vectors.

The repo already has `server-fixture` and `harness` crates, and the server fixtures use `futures-rustls`, so you have a natural place to host TLS 1.3-only local endpoints for deterministic regression tests. RFC 8448 then gives you reproducible traces with private keys, intermediate secrets, traffic keys, and IVs, including a simple 1-RTT handshake, resumed 0-RTT, HelloRetryRequest, and client authentication.  [oai_citation:15‡GitHub](https://raw.githubusercontent.com/tlsnotary/tlsn/main/Cargo.toml)

Minimum green bar:

- HKDF unit tests match RFC-derived expectations
- simple 1-RTT TLS 1.3 works against a local fixture
- prover/verifier can commit and selectively disclose application data over TLS 1.3
- existing TLS 1.2 tests still pass
- HRR passes before X25519 lands

## One hard rule

Do **not** reveal application traffic secrets just because you revealed handshake secrets.

Revealing only handshake keys is what keeps the TLS 1.3 port practical: RFC 8446 encrypts the server handshake after `ServerHello`, and DECO’s TLS 1.3 adaptation uses that separation to let the prover decrypt the server handshake locally without giving up the confidentiality of the final application session keys.  [oai_citation:16‡IETF Datatracker](https://datatracker.ietf.org/doc/html/rfc8446)

The clean starting point is: open the TLS 1.3 epic against `#859`, make `#978` + `#1001` the first concrete workstream, and treat **“1-RTT, P-256, AES-128-GCM-SHA256, no PSK”** as the first definition of done.
